# GATES · 门禁评分台（THE GRADER）【开发os级别】

> 拿到任何改动,**先来这里定门禁**:落到哪一级风险 → 必备工件 + 必跑测试是哪套。
> 风险阶梯是这套 OS 的单一权威源 `[权威]`;别处(RULES / 模板 / autonomy)只用一句指针引本文,不复述。
> **触发哪一级**(碰什么算高风险)是项目内容 → 全留 `<...>` 占位,填在 `RULES.project.md`。

## 0. 怎么用（改动先来这定门禁）

每个改动(PR / 任务 / 一回合实现)进实现前走三步:

1. **定级**:对照 §1 阶梯,把改动落到 R0–R5 之一。**先看 §2 升级地板**——碰了敏感面即抬级,不因「改得小」降级;不确定取高。
2. **取工件 + 测试**:照该级「必备工件集 + 必跑测试类」备齐;缺一不算到位。高风险工件用 `tasks/_templates/` 仓库模板(见 `HARNESS_INDEX.md`),不自造格式。
3. **打标 + 自述**:在 PR / 任务卡 header 标 `risk_level=R?`(**未打标不可合**,见 §2),交付时按 §6 出可审计自述清单。

> 风险级也回写任务卡 header 的 `risk_level` 槽(见 `tasks/_templates/TASK.md`)与 `BOARD.md` 的「风险级」列,让取卡时一眼可见。

## 1. 风险阶梯 R0–R5 `[权威]`

每级 = **一句话语义** + **必备工件集** + **必跑测试类**。「触发类别」列是项目内容,留 `<...>` 占位,枚举填在 `RULES.project.md` §风险阶梯触发条件。

| 级 | 一句话语义 | 触发类别（PROJECT-FILL → RULES.project） | 必备工件集 | 必跑测试类 |
|---|---|---|---|---|
| **R0** | 无行为变化的纯文本 | 错别字 / 注释 / 无逻辑变化的静态文案 `<...>` | PR 说明标 `risk_level=R0` | 无（声明无行为变化即可） |
| **R1** | 低风险展示层 | 非业务 UI / 静态布局 / 局部样式 `<...>` | 轻档说明（`LIGHT_TSD` §低风险证明全-no） | 截图 **或** 明确「无需测试」理由（写进 TestEvidence 的 `not_run_reason`） |
| **R2** | 普通业务对象 / 接口 | 普通 API / 普通数据对象（**非**`<敏感面占位:钱/信用/证据…>`）`<...>` | 接口/数据契约同步（§4 硬同步） | 单测 **或** 契约测试（TestEvidence 据实填） |
| **R3** | 碰核心契约面 | 状态机 / 权限 / 正式事实命令 / 证据 schema / `<Agent 输出 schema 等占位>` / 后台审核 / 验收 `<...>` | `TSD` + 状态机/权限说明 + 反引 task/test id | 针对失效模式的对抗测试 + 变形 + 幂等（RULES §2）；建议二次评审**留痕**(非合并门) |
| **R4** | 碰不可逆 / 资金 / 终局裁决 | 资金 / 信用硬事件 / 争议终决 / 禁止动作边界 / 数据安全 / 生产迁移 `<...>` | `ADR` + `TSD` + 完整测试 + 回滚/补偿计划 + release gate | R3 全部 + 回滚演练 + 补偿/幂等证据；建议二次评审 |
| **R5** | 系统级治理 / 生产学习边界 | 跨主体 / 运行日志 / 训练数据 / 批量演化 / 发布闸 / 后台控制台 / `<生产业务后果边界占位>` `<...>` | R4 全部 + TestEvidence + release/冻结方案齐备 | R4 全部 + 多视角二次评审建议；运行时仍受 **§6 不可逆真钱人审门** 约束（工程去门禁**不放开**它） |

> 工程过程门(本表 R0–R5 的工件/测试)按双轴属「可降级为可审计」轴(见 §6 + RULES §双轴门禁边界);**唯一永不降**的是 R4/R5 落到不可逆真钱/真信用/生产 flag 时的运行时人审(双签)——那是另一条轴,见 §6。

## 2. 升级地板（碰即抬级 · 不确定取高 · 未打标不可合）

三条硬规则,任何定级都先过它们:

1. **碰敏感面即抬级（地板，不能因改得小而降级）**——占位表填在 `RULES.project.md`:

   | 碰到（PROJECT-FILL → RULES.project §风险阶梯触发条件） | 地板 |
   |---|---|
   | `<影响正式业务事实/权限/证据/合同/验收/争议/后台裁决/Agent 边界…占位>` | **≥ R3** |
   | `<资金释放/保证金冻结扣划/信用硬事件/数据泄露风险/生产迁移…占位>` | **≥ R4** |
   | `<跨主体治理/生产学习/发布闸/正式业务后果边界…占位>` | **≥ R5** |

   > 怎么填:把项目里「碰了就不许当低风险」的面逐类枚举进 RULES.project,每类钉到上面对应地板。这张表是 §1「触发类别」列的权威枚举家;§1 只引不复述。

2. **不确定取高**:判不准落哪级 → 按**更高**级处理,并在 PR/任务卡写清判断依据。

3. **未打标不可合**:任何 PR/任务/一回合实现**必须**带 `risk_level=R?`;没打标 = 门禁未定 = 不可合(`validate_*` 兜底查)。

## 3. DoR / DoD 双闸（进实现前 · 算完成前）

两道 checklist 骨架,通用可复用;具体阈值/产物名按项目填。

### DoR — 进实现前（Definition of Ready）
进 `tasks/active/` 写代码前,全勾才动手:
- [ ] 已定级 `risk_level=R?`,并过了 §2 三条升级地板。
- [ ] 该级 §1「必备工件集」已识别(高风险已选好 `tasks/_templates/` 模板)。
- [ ] 待拍板项清零——任务卡 `Open Questions` 无 `[需拍板]`(计数 `（已决 D/总）` 满格);有待拍 → 先停、请人拍(RULES §7)。
- [ ] 接线点 / Scope / 验收口径 已写进任务卡(对应节非空,非 `<...>`)。
- [ ] 上游事实源无未决冲突(§5);有冲突先上报改上游再开工。

### DoD — 算完成前（Definition of Done）
落 `tasks/done/` + 标「done」前,全勾才算完成:
- [ ] §1 该级「必跑测试类」已跑,**未破坏现有基线**(数目以实跑为准,别写死)。
- [ ] 对抗测试到位:「种一个已知坏,门必抓」(RULES §2)——不是覆盖率。
- [ ] 工件齐 + 互引 id 接好(task↔TSD↔review↔test↔ADR,见模板 header-as-wiring)。
- [ ] §4 文档同步:契约类**同 commit 硬同步**已完成;导航类 best-effort。
- [ ] §6 可审计自述清单已出(改了啥 / 测了啥 TestEvidence / 决策依据)。
- [ ] STATE 刷新(诚实标 🟡未验证 ≠ ✅;带 provenance);`validate_dev.py` 自检过。
- [ ] **若碰不可逆真钱门**:运行时执行**未**被自动放行——留人审(双签)入口,AI 不自动执行(§6)。

## 4. change → doc + test 同步矩阵（两速）

文档同步**分两速**,别用一个速度齐刷:

| 速度 | 文档类 | 时机 | 失败后果 |
|---|---|---|---|
| **硬同步** | **契约类**:`<正式事实/状态机/权限/API契约/DB设计/Agent契约… 占位>` | **同 commit** 必须改完 | 不同步 = 改动不可合(契约与代码漂移) |
| **软同步** | **导航类**:索引 / TRACE / 进度台 / BOARD 摘要 `<...>` | best-effort,允许稍后补 | 漂了修索引、**以原文为准**(§5),不阻塞合并 |

测试同步:按 §1 该级「必跑测试类」走;行为变化必补测试,结论走 `TEST_EVIDENCE` 模板(禁裸写「passed/通过」)。

> 怎么填:把项目里「改了它就必须同 commit 改契约文档」的清单填进 `RULES.project.md`(契约类),其余索引/进度类归软同步。契约类 vs 导航类的判据 = §5 事实源层级:能反定义产品事实的是契约类(硬),只导航的是导航类(软)。

## 5. 事实源层级 + stop-on-conflict `[权威]`

**下层永不反定义上层**;冲突即停、上报、改上游再继续。层级骨架(具体文档名 PROJECT-FILL):

```
产品事实源（PRD / 需求契约）           ← 最高,定义产品事实
  └─ 跨模块契约（正式事实/状态机/权限/证据/Agent 契约）
       └─ 接口契约（API / OpenAPI）
            └─ 持久化契约（DB 设计）
                 └─ 代码 / 实现           ← 最低,不得反向定义上面任何一层
导航类（索引 / TRACE / 进度台）= 旁路:只定位,不进层级,冲突以被指向原文为准
```

铁律:
- **现有代码不能反向定义业务契约**;ADR 可覆盖旧**工程**方案,但不能覆盖未同步更新的业务事实/状态机/权限/资金/信用/证据规则。
- **导航/索引文件**(本 GATES 之外的 `HARNESS_INDEX` / 各 INDEX / TRACE)不能推翻被指向原文;索引与原文冲突 → **先修索引**。
- **stop-on-conflict**:发现任意两源冲突 → **停止相关实现**,输出冲突清单,**先更新上游契约**,再继续。别在下层「就地和稀泥」把上层悄悄改了。

> 怎么填:把项目真实的文档名挂到上面每层(PRD 名 / 契约文档名 / API spec / DB 设计),填在 `RULES.project.md`;本节只给层级**机制**,不收项目文档名。

## 6. 可审计取代评审（每交付自述清单 · 不可逆真钱人审门永不降）

工程过程侧**不再**强制人工评审作合并门禁(双轴的「可降级」轴);改以**可审计**为准。每次交付出一份自述清单:

- **改了啥**:改了哪些文件 / 文档(契约类是否同 commit 同步,§4)。
- **测了啥**:跑了哪些测试,走 `TEST_EVIDENCE`(`command`/`cwd`/`exit_code`/`status`/`output_summary` **或** `not_run_reason`+回填 owner);禁裸写「通过」。
- **决策依据**:静默决策记 `exec/GAP_LOG.md`(GAP-NNNN);架构取舍记 `ADR`。
- **二次评审**:R3+ **建议**做一轮(人工或工具)并留痕,但**不再是合并门禁**;agent 可起草不可自判 valid(见 `REVIEW_RECEIPT` 反自批)。

**双轴边界(永不降的那条)** —— 与 RULES §双轴门禁边界、`autonomy/LOOP_CONTRACT.md` 措辞一致:

> ⛔ **不可逆真钱人审门永不降级**。R4/R5 落到 **真钱 / 真信用 / 不可逆动作 / 真争议裁决 / 生产 flag / 真 KMS / 对外发布**（权威清单见 `RULES.md §8`）时,运行时执行**永远人审(双签)**,**AI 不得自动执行**。工程过程门去门禁(改可审计)**不波及**这条——它是另一条正交轴(运行时不可逆门),无论改动可审计与否都拦在放行前。
> 这是本 OS 最值钱的自治元规则:可审计取代的是**评审签字**,**不是**不可逆放行的人类双签。
